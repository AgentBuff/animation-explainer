SCENE 01 · 起点

一个 .vue 文件

Vue 的旅程从一个 .vue 文件开始 — SFC（Single File Component）把模板、逻辑、样式装进一个文件。 但浏览器只认 HTML/CSS/JS，这三块怎么变成屏幕上能动的 UI？接下来 7 节，从编译到渲染，逐层揭开。

<template> <div class="counter"> <h1>Count: {{ count }}</h1> <button @click="count++">+1</button> </div> </template> <script setup> import { ref } from 'vue' const count = ref(0) </script>
SCENE 02 · 编译期

Template → Render Function

模板不是运行时解析的，而是编译期就被转成 render() 函数。流程： 源码 → AST → Transform（这一步做静态提升、加 patchFlag、生成 blockTree）→ 代码生成。 编译器在这一刻就已经"看懂"了哪些节点是静态的、哪些会变 —— 运行时根本不需要再判断。

// 静态节点被提升出 render 函数，全局只创建一次 const _hoisted_1 = _createElementVNode("button", null, "+1") export function render(_ctx) { return (_openBlock(), _createElementBlock("div", { class: "counter" }, [ _createElementVNode("h1", null, "Count: " + _toDisplayString(_ctx.count), 1 /* TEXT */ // ← patchFlag: 只有文本会变 ), _hoisted_1 // ← 复用静态节点 ])) }
SCENE 03 · 响应式核心

reactive() 给对象套上 Proxy

reactive(obj) 不是返回新对象的复制，而是返回一个 Proxy —— 一个能拦截所有读写的"代理层"。 原始对象毫发无损，但用户对 Proxy 的每次 get / set 都会被拦截器接住， 这正是 Vue 能"感知"数据变化的根本机制。

// packages/reactivity/src/reactive.ts export function reactive(target) { return new Proxy(target, { get(t, key) { track(t, key) // 收集依赖 return Reflect.get(t, key) }, set(t, key, value) { Reflect.set(t, key, value) trigger(t, key) // 触发更新 } }) }
SCENE 04 · 收集

track：建立"谁依赖谁"的图

当 effect（渲染函数、computed、watchEffect 都是 effect）运行时，每次访问响应式属性， Proxy 的 get 拦截器就调用 track，把当前 effect 加进一个三层 Map： WeakMap<target → Map<key → Set<effect>>>。这就是 Vue 知道"state.count 改了，谁要重跑"的依据。

let activeEffect // 当前正在运行的 effect（全局唯一） const targetMap = new WeakMap() function track(target, key) { if (!activeEffect) return let depsMap = targetMap.get(target) if (!depsMap) targetMap.set(target, (depsMap = new Map())) let dep = depsMap.get(key) || depsMap.set(key, new Set()).get(key) dep.add(activeEffect) // "我在依赖这个 key" }
SCENE 05 · 触发

trigger：通知所有受影响的 effect

当 state.count++ 触发 set 拦截器，trigger 反查刚才建立的图： 找到这个 key 对应的 effect 集合，全部重跑。渲染函数也是一个 effect —— 所以一行赋值，UI 就跟着更新。 整个机制不需要 dirty checking，也不需要 setState：依赖关系本身就是"自动追踪"的。

function trigger(target, key) { const depsMap = targetMap.get(target) if (!depsMap) return const dep = depsMap.get(key) if (!dep) return dep.forEach(effect => { if (effect.scheduler) effect.scheduler() // 异步批量更新 else effect.run() // 同步重跑 }) } // 多个 set 在同一 tick 内会被 scheduler 合并，只重跑一次渲染
SCENE 06 · 虚拟 DOM

render() 产出 VNode 树

render 函数运行的结果不是 DOM，而是 VNode 树 —— 一棵用普通对象描述 UI 的轻量级镜像。 为什么不直接操作 DOM？因为对比两个 JS 对象比对比两棵 DOM 树快得多， 而且 VNode 还能附带 patchFlag、shapeFlag、key 这些编译期/运行期的元信息，让后续 diff 更高效。

// VNode 的完整 shape { type: "div" | Component | Symbol, props: { class, style, onClick, ... } | null, children: VNode[] | string | null, key: string | number, // diff 复用的依据 patchFlag: 1, // 1=TEXT 2=CLASS 4=STYLE 8=PROPS shapeFlag: 9, // 节点+children 形态的位运算掩码 el: HTMLElement | null, // 挂载后指向真实 DOM component: ComponentInstance | null, dynamicChildren: VNode[] // blockTree 跳过静态节点 }
SCENE 07 · Diff

新旧 VNode 比对

当一个组件重渲染，Vue 不会重建整棵 VNode 对应的 DOM。 Diff 算法在同层用 key 找到能复用的节点 — 相同 key 就 patch、位置变了就移动、新增的才 createElement、旧的去掉就 removeChild。 复杂度 O(n)，再用最长递增子序列把"移动"的次数降到最少。

// packages/runtime-core/src/renderer.ts · patchKeyedChildren function patchKeyedChildren(c1, c2, container) { let i = 0, e1 = c1.length - 1, e2 = c2.length - 1 // 1. 从头开始 sync：A == A → patch，继续 while (i <= e1 && i <= e2 && isSameVNodeType(c1[i], c2[i])) { patch(c1[i], c2[i], container) i++ } // 2. 从尾开始 sync（同理） // 3. 中间乱序：建立 key→index 映射，找最长递增子序列，最少移动 // 4. 多余的旧节点 removeChild；多余的新节点 createElement }
SCENE 08 · 落地

Patch：VNode → 真实 DOM API

最后一公里：renderer 把 VNode 翻译成浏览器认识的 DOM 操作。 首次挂载是一连串 createElement / setAttribute / appendChild； 更新时则只动会变的节点 —— 拜 patchFlag 所赐，h1.textContent = "Count: 1" 一句就够了， 整棵树其余部分纹丝不动。这就是 Vue 性能的来源：编译期标注 + 运行时精准更新。

// packages/runtime-dom/src/nodeOps.ts (浏览器平台的实现) export const nodeOps = { createElement: (tag) => document.createElement(tag), setText: (node, text) => { node.nodeValue = text }, insert: (child, parent, anchor) => parent.insertBefore(child, anchor), remove: (child) => child.parentNode?.removeChild(child), setElementText:(el, text) => { el.textContent = text }, patchProp: (el, key, prev, next) => { /* 处理 class/style/event/attr */ } } // runtime-core 完全平台无关 — 换 nodeOps 就能渲染到原生 / Canvas / SSR
