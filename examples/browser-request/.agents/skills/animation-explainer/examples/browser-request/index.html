SCENE 01 · 起点

你按下了回车键

一切从一行字开始：https://example.com。 它看起来只是浏览器地址栏里的一段文本，但按下回车的瞬间，背后正在动员一整套互联网基础设施 —— DNS 服务器、路由器、操作系统、Web 服务器、数据库，最后才是你眼前的像素。

https://example.com:443/ # ┬ ┬ ┬ ┬ # │ │ │ └─ 路径 path # │ │ └───── 端口 port（HTTPS 默认 443） # │ └─────────────────── 主机名 hostname（需要先翻译） # └─────────────────────────── 协议 scheme
SCENE 02 · 名字翻译

DNS：把名字变成地址

互联网底层只认 IP 不认名字。浏览器先问本地 DNS Resolver， Resolver 一级级向上：Root → .com TLD → example.com 权威服务器， 最终拿到 93.184.216.34，然后缓存起来。

$ dig example.com +short 93.184.216.34 $ dig example.com ;; ANSWER SECTION: example.com. 86400 IN A 93.184.216.34 ;; Query time: 12 msec
SCENE 03 · 建立连接

TCP 三次握手

有了 IP 还不够。TCP 是可靠的传输协议 —— 在发数据前要先建立一条"连接"。 流程像两个人见面打招呼三次：客户端说 SYN，服务器回 SYN+ACK，客户端再 ACK 确认。 至此双方都确认了对方能收发数据，连接正式建立。

# tcpdump trace 10:23:01.001 Client → Server Flags=SYN seq=1000 10:23:01.024 Server → Client Flags=SYN,ACK seq=5000 ack=1001 10:23:01.025 Client → Server Flags=ACK ack=5001 # RTT ≈ 24ms · 连接已建立
SCENE 04 · 上锁

TLS 握手：建立加密通道

因为是 https://，在 TCP 之上还要做一次 TLS 握手。 浏览器先发 ClientHello 告诉服务器自己支持的密码套件； 服务器回 ServerHello 选定算法、附上 证书；浏览器验证证书后双方协商出一个对称密钥。 之后所有 HTTP 报文都用这把密钥加密。

# 证书验证：浏览器检查三件事 1. 签名链 : 证书是不是受信 CA 签发的？ 2. 域名匹配: 证书里的 CN/SAN 是不是 example.com？ 3. 有效期 : notBefore ≤ now ≤ notAfter # 任何一项失败 → 浏览器弹出"不安全"警告
SCENE 05 · 发出请求

HTTP 请求：一封纯文本"信"

很多人以为 HTTP 神秘，其实它就是一段格式化的纯文本。 第一行写动词和路径，后面跟若干 Header: Value，最后空一行 —— 就这么简单。 GET 请求通常没有 body；POST/PUT 才在空行后跟正文。

GET / HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X) Accept: text/html,application/xhtml+xml Accept-Language: zh-CN,zh;q=0.9 Accept-Encoding: gzip, deflate, br Connection: keep-alive # 空行表示 header 结束，GET 没有 body
SCENE 06 · 服务器内部

请求在服务器里的旅程

报文到了服务器，先被 反向代理（Nginx）接住、做 TLS 终止； 然后转给应用进程，路由器匹配 URL 找到对应的 handler； handler 执行业务逻辑，可能去 数据库 取数据、渲染模板，最后把响应交回去。

[2026-05-17 10:23:01.027] → nginx[1]: proxy_pass to upstream :3000 → rails[3000]: Started GET "/" for 203.0.113.5 → rails[3000]: Routed to HomeController#index → rails[3000]: SELECT * FROM pages WHERE slug='home' (3ms) → rails[3000]: Rendered home.html.erb (12ms) ✓ rails[3000]: Completed 200 OK in 18ms
SCENE 07 · 回程

HTTP 响应：返程的信

响应的结构和请求几乎对称 —— 第一行是状态码（200/404/500…）， 后面是头部（告诉浏览器内容类型、长度、缓存策略），空行之后才是真正的 body。 浏览器拿到这块文本，旅程进入最后一段：渲染。

HTTP/1.1 200 OK Content-Type: text/html; charset=utf-8 Content-Length: 1432 Cache-Control: public, max-age=3600 Date: Sun, 17 May 2026 10:23:01 GMT <!DOCTYPE html> <html><head><title>Example</title></head> <body>Hello!</body> </html>
SCENE 08 · 终点

浏览器把字节变成像素

浏览器把 HTML 解析成 DOM 树，CSS 解析成 CSSOM 树，二者合并成 Render Tree（只包含会渲染的节点）。 接着 Layout 算出每个盒子的位置和尺寸，Paint 涂色并切分图层，最后由 GPU Composite 合成 —— 像素终于出现在屏幕上。整段旅程通常不到 200ms。
